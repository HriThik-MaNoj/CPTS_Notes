## Claude notes
# ACL Abuse ‚Äî Quick Reference

## What is ACL Abuse?

Active Directory objects have Access Control Lists (ACLs) defining _who can do what_ to them. When an attacker controls a user with excessive ACL rights over another object, they can escalate privileges through misconfigured permissions alone ‚Äî no exploit needed.

---

## Key Rights

|Right|What it lets you do|
|---|---|
|`ForceChangePassword`|Reset a user's password without knowing the current one|
|`GenericWrite`|Write any non-protected attribute (group members, SPNs, etc.)|
|`GenericAll`|Full control ‚Äî includes all rights above|

---

## Attack Chain

```
wley ‚îÄ‚îÄ[ForceChangePassword]‚îÄ‚îÄ‚ñ∂ damundsen ‚îÄ‚îÄ[GenericWrite]‚îÄ‚îÄ‚ñ∂ Help Desk L1
‚îÄ‚îÄ[nested]‚îÄ‚îÄ‚ñ∂ IT Group ‚îÄ‚îÄ[GenericAll]‚îÄ‚îÄ‚ñ∂ adunn (admin)
```

**Why it works:** You never directly attack `adunn`. Each hop uses a legitimate AD permission. No CVE, no exploit.

**Nested group trick:** Adding `damundsen` to `Help Desk Level 1` silently inherits IT Group's rights because `Help Desk L1` is nested inside it.

**Why Kerberoast instead of resetting adunn's password:** `adunn` is an admin ‚Äî changing its password locks out the real user and alerts defenders. Kerberoasting grabs the hash silently; `adunn` keeps working normally while you crack offline.

---
