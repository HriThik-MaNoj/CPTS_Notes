# 🛡️ Enumerating Security Controls

> [!abstract] Module Summary After gaining a foothold, enumerate the defensive posture of hosts to understand what tools/techniques to use or avoid. Security controls vary per machine — some may be more locked down than others.

**Tags:** #activedirectory #enumeration #security-controls #post-exploitation #pentesting

---

## 🔑 Why This Matters

- Security tools in use affect **which enumeration tools** you can run
- Knowing defenses helps you **plan around or modify** your approach
- Controls are **not always applied uniformly** across an org — look for gaps

---

## 1. 🪟 Windows Defender

> [!info] What It Is Microsoft's built-in AV/EDR. Greatly improved — will block tools like **PowerView** by default.

### Check Defender Status

```powershell
Get-MpComputerStatus
```

### Key Fields to Check

|Field|Meaning|
|---|---|
|`RealTimeProtectionEnabled`|Is live scanning on?|
|`AntivirusEnabled`|AV engine active?|
|`BehaviorMonitorEnabled`|Behavioral detection on?|
|`AMServiceEnabled`|Core service running?|

> [!tip] Bypass Bypasses exist — covered in separate modules.

---

## 2. 🔒 AppLocker

> [!info] What It Is Microsoft's **application whitelisting** solution. Controls which executables, scripts, DLLs, and installers can run.

### Check Effective Policy

```powershell
Get-AppLockerPolicy -Effective | select -ExpandProperty RuleCollections
```

### Common Misconfigurations to Look For

> [!warning] Common Bypass Opportunity Orgs often block `powershell.exe` but forget **alternate locations**:
> 
> - `%SystemRoot%\SysWOW64\WindowsPowerShell\v1.0\powershell.exe`
> - `PowerShell_ISE.exe`

### Reading AppLocker Output

|Field|What to Check|
|---|---|
|`PathConditions`|Which paths are blocked/allowed|
|`Action`|`Deny` or `Allow`|
|`UserOrGroupSid`|Who the rule applies to|
|`Name`|Rule description|

**Example — Blocked:**

```
PathConditions : {%SYSTEM32%\WINDOWSPOWERSHELL\V1.0\POWERSHELL.EXE}
Action         : Deny
UserOrGroupSid : <Domain Users SID>
```

→ Only the standard path is blocked. Other PS locations likely still work.

## 3. 🔐 PowerShell Constrained Language Mode

> [!info] What It Is Locks down PowerShell features — blocks COM objects, restricts .NET types, disables PS classes, etc.

### Check Language Mode

```powershell
$ExecutionContext.SessionState.LanguageMode
```

### Possible Values

|Value|Meaning|
|---|---|
|`FullLanguage`|No restrictions — you're good|
|`ConstrainedLanguage`|Heavy restrictions — many tools will break|

> [!danger] Impact If `ConstrainedLanguage` is set, tools like PowerView, BloodHound ingestors, etc. may fail or need workarounds.

## 4. 🔑 LAPS (Local Administrator Password Solution)

> [!info] What It Is Randomizes & rotates local admin passwords on Windows hosts to **prevent lateral movement**. If you can read LAPS passwords, you get local admin on those machines.

### Tool: LAPSToolkit

#### 1. Find Who Can Read LAPS Passwords

```powershell
Find-LAPSDelegatedGroups
```

→ Shows OUs + delegated groups (usually `Domain Admins` + `LAPS Admins`)

#### 2. Find Users with "All Extended Rights"

```powershell
Find-AdmPwdExtendedRights
```

> [!tip] Why This Matters Users with **All Extended Rights** can read LAPS passwords but may be less protected than delegated group members — good targets to look for.

#### 3. Read LAPS Passwords (if you have access)

```powershell
Get-LAPSComputers
```

→ Returns `ComputerName`, `Password` (cleartext!), and `Expiration`

**Example Output:**

```
ComputerName                Password       Expiration
DC01.INLANEFREIGHT.LOCAL    6DZ[+A/[]19d$F 08/26/2020 23:29:45
```

## 🗺️ Quick Reference Cheatsheet

|Control|Detection Command|What to Look For|
|---|---|---|
|**Defender**|`Get-MpComputerStatus`|`RealTimeProtectionEnabled = True`|
|**AppLocker**|`Get-AppLockerPolicy -Effective`|Blocked paths, alternate PS locations|
|**PS Language Mode**|`$ExecutionContext.SessionState.LanguageMode`|`ConstrainedLanguage` = bad|
|**LAPS Groups**|`Find-LAPSDelegatedGroups`|Which OUs/groups manage LAPS|
|**LAPS Extended Rights**|`Find-AdmPwdExtendedRights`|Users with all extended rights|
|**LAPS Passwords**|`Get-LAPSComputers`|Cleartext passwords if access|

---

## 🧠 Key Takeaways

> [!success] Things to Remember
> 
> 1. **Enumerate before you tool up** — knowing defenses saves you from burning your access
> 2. **AppLocker ≠ total lockdown** — always check alternate PS paths
> 3. **LAPS users with extended rights** are high-value targets — less guarded than delegated groups
> 4. **Controls are uneven** — a locked workstation ≠ a locked server
> 5. Bypassing these controls is out of scope here — separate modules cover it
