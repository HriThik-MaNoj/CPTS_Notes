### Disabling Front-end Validation

#### Concept

Because client-side validation runs entirely within the user's browser, the execution environment is fully controllable. Scripts can be modified, bypassed, or deleted using browser developer tools, eliminating the need to intercept and modify requests using tools like Burp Suite.

#### Methodology

1. **Identify the Target Input Element**
    
    - Open the Page Inspector: `Ctrl + Shift + C` (Windows/Linux) or `Cmd + Option + C` (Mac).
        
    - Click on the file upload button or trigger zone to locate the corresponding HTML code.
        
    - Look for restrictive attributes or event listeners:
        
        HTML
        
        ```
        <input type="file" name="uploadFile" id="uploadFile" onchange="checkFile(this)" accept=".jpg,.jpeg,.png">
        ```
        
2. **Analyze the Validation Script (Optional)**
    
    - Open the Web Console: `Ctrl + Shift + K` (Windows/Linux) or `Cmd + Option + K` (Mac).
        
    - Type the name of the validation function (e.g., `checkFile`) and press `Enter` to view its implementation details and logic.
        
3. **Disable the Client-Side Restrictions**
    
    - Return to the **Inspector/Elements** tab.
        
    - Double-click the event handler attribute executing the validation logic (e.g., `onchange="checkFile(this)"`) and delete it entirely.
        
    - _(Optional)_ Delete the `accept="..."` attribute to display all file types by default in the file selection dialog.
        
    - _Note:_ These modifications are temporary and live only in the current browser session memory; refreshing the page restores the original restrictions.
        
4. **Upload and Execute**
    
    - Select and upload the desired file type (e.g., a PHP web shell) via the modified form UI.
        
    - Use the Page Inspector again to find the rendering path or source URL of the uploaded asset.
        
    - Access the uploaded file directly to interact with the payload:
        
        Code snippet
        
        ```
        http://SERVER_IP:PORT/profile_images/shell.php?cmd=id
        ```

## DoS
- we can utilize aô `Decompression Bomb`ô with file types that use data compression, likeô `ZIP`ô archives.
- Another possible DoS attack is aô `Pixel Flood`ô attack with some image files that utilize image compression, likeô `JPG`ô orô `PNG`. We can create anyô `JPG`ô image file with any image size (e.g.ô `500x500`), and then manually modify its compression data to say it has a size of (`0xffff x 0xffff`), which results in an image with a perceived size of 4 Gigapixels. When the web application attempts to display the image, it will attempt to allocate all of its memory to this image, resulting in a crash on the back-end server.


## Other Upload Attacks
- Other than arbitrary file uploads and limited file upload attacks, there are a few more techniques.

## Injections in File Name
For example, if we name a fileô `file$(whoami).jpg`ô orô ``file`whoami`.jpg``ô orô `file.jpg||whoami`, and then the web application attempts to move the uploaded file with an OS command (e.g.ô `mv file /tmp`), then our file name would inject theô `whoami`ô command, which would get executed, leading to remote code execution.
- we may use an XSS payload in the file name (e.g.ô `<script>alert(window.origin);</script>`), which would get executed on the target's machine if the file name is displayed to them.
- We may also inject an SQL query in the file name (e.g.ô `file';select+sleep(`

## Upload Directory Disclosure
- we may utilize fuzzing to look for the uploads directory or even use other vulnerabilities (e.g., LFI/XXE) to find where the uploaded files are by reading the web applications source code.
- Try to trigger error messages which may expose the uploads directory
	- By trying an already existing filename.
	- By trying a really long filename (eg 5000 characters)

## Windows-specific Attacks
- using reserved characters, such as (`|`,ô `<`,ô `>`,ô `*`, orô `?`), which are usually reserved for special uses like wildcards. If the web app doesn't properly sanitize these inputs, it may cause an error that discloses the upload directory.
- Similarly, we may use Windows reserved names for the uploaded file name, like (`CON`,ô `COM1`,ô `LPT1`, orô `NUL`), which may also cause an error as the web application will not be allowed to write a file with this name.
- ô we may utilize the Windowsô [8.3 Filename Convention](https://en.wikipedia.org/wiki/8.3_filename)ô to overwrite existing files or refer to files that do not exist.
- Older versions of Windows were limited to a short length for file names, so they used a Tilde character (`~`) to complete the file name, which we can use to our advantage.For example, to refer to a file called (`hackthebox.txt`) we can use (`HAC~1.TXT`) or (`HAC~2.TXT`), where the digit represents the order of the matching files that start with (`HAC`). As Windows still supports this convention, we can write a file called (e.g.ô `WEB~1.CON`) to overwrite theô `web.conf`ô file.
